# Audio-Spectrum-Visualizer



## Overview
---

This project implements a real-time audio spectrum visualizer for RP2040 using:

- DMA ADC capture
- FFT (kissFFT)
- multicore processing
- WS2812B LED matrix rendering

Audio spectrum is visualized as colored concentric rectangles.
---

Обчислення спектра аудіосигналу та відображення його на RGB LED-матриці **WS2812B 16×16** (NeoPixel) у вигляді концентричних квадратів, де розмір відповідає рівню сигнала, а колір — спектральним складовим. 

---

## 📦 Основні характеристики

### Відображення

- Діапазон частот, який відображає індикатор, визначається параметрами:
  - `Fs` / `SAMPLE_FREQ` — частота семплування (Гц)
  - `N` / `FFT_SIZE` — розмір FFT
  - `IND_BANDS` — розбиття FFT-бінів на смуги (які біни реально візуалізуються)

(див. розділ “Розрахунок діапазону частот”)

- **8 частотних кольорів**. Кожний колір відповідає за певний частотний діапазон 
- **8 рівнів (0..8)** сигналу (обмежується розміром матриці), масштабування від `dBFS`
- Динамічний масштаб **AGC по загальному рівню**
- Визначаються домінантні частотні діапазони та відображаються за їх рівнем та кольором у відповідних концентричних квадратах
- Максимальна частота оновлення індикації: **~10 FPS**

### FFT

- Розмір: **N = 1024**
- Вікно: **Hann window** (вікно Ханна), **Hamming window** (Вікно Геммінга), або без вікна
- Реалізація FFT: **kissFFT (float)** + обчислення енергії в C через модуль `fastfft`  
  Репозиторій: https://github.com/Alex-Teteria/Fast-FFT-module-for-MicroPython-RP2040-

### ADC (RP2040)

- Вибір частоти семплування `Fs` робиться під роздільну здатність частотних смуг  
  `Δf = Fs / FFT_SIZE`
  (див. розділ “Розрахунок діапазону частот”) 
- Захват: C-модуль `adc_dma` (one-shot/burst, апаратне тактування, DMA у RAM)  
  Репозиторій: https://github.com/Alex-Teteria/MicroPython-RP2040-ADC-DMA-Extension
- Окремий аналоговий антиаліасовий фільтр можливо не застосовувати; вхідний тракт (електретний мікрофон + мікрофонний підсилювач) призначений для аудіо й має обмежену корисну смугу. Крім того, вплив ВЧ артефактів обмежений вибором частотного діапазону для відображення.

## 🏗️ Апаратна частина та підключення

### Модулі

- MAX4466 або MAX9814, electret mic amplifier module (мікрофон + підсилювач)
- WS2812B RGB LED matrix 16×16 (256 LED)
- Raspberry Pi Pico / RP2040

### Живлення та сигнали

- MAX4466 (MAX9814) живити від **3.3V** Pico.
- Вихід MAX4466 має **DC offset ~VCC/2**, MAX9814 має **DC offset ~1,25В** тому можуть подаватись напряму на ADC.
- LED-матрицю WS2812B живити від **5V окремим джерелом** (струм залежить від яскравості та кольорів).
- Обов’язково зробити **спільну землю (GND)** між Pico та 5V LED-джерелом.
- DATA лінія WS2812B:
  - підключити послідовно резистор **330–470 Ω** у DATA біля входу матриці.

### Піни

- ADC: GPIO26 / ADC0
- NeoPixel DATA: GPIO20 (відповідає `neo_pin=20` у коді)

## :triangular_ruler: Розрахунок діапазону частот

#### Крок по частоті (роздільна здатність FFT)

Для real FFT з розміром `N` частотний крок одного біну:

$$
\Delta f = \frac{Fs}{N}
$$

#### Відповідність "бін → частота"

Частота біну з індексом `k`:

$$
f(k) = k \cdot \Delta f = k \cdot \frac{Fs}{N}
$$

де:

- `k = 0` — DC (0 Гц), не візуалізується
- верхня межа спектра обмежена частотою Найквіста:

$$
f_{Nyq} = \frac{Fs}{2}
$$

#### Діапазон відображення

Якщо фактично візуалізуються біни від `k_min` до `k_max` (включно), то приблизний діапазон відображення:

$$
f_{min} \approx k_{min} \cdot \frac{Fs}{N}, \quad
f_{max} \approx k_{max} \cdot \frac{Fs}{N}
$$

Значення `k_min` і `k_max` визначаються реальною областю бінів, яку покриває `IND_BANDS`
(тобто найменший/найбільший бін, що входить у будь-яку смугу).

#### Приклад

Для `Fs = 40_000` і `N = 1024`:

$$
\Delta f \approx \frac{40\,000}{1024} \approx 39.06 \text{ Гц}
$$

Якщо `IND_BANDS` охоплює біни `k_min = 1` … `k_max = 51`, тоді:

- $f_{min} \approx 1 \cdot 39.06 \approx 39$ Гц
- $f_{max} \approx 51 \cdot 39.06 \approx 1992$ Гц

> Примітка: реальні межі можуть відрізнятись залежно від того, які саме біни/смуги включені у `IND_BANDS`.

## :hammer: Приклад практичної реалізації

Використано модуль MAX9814 (мікрофон + підсилювач), gain=40dB      

### Вхідні параметри

- `Fs = 40_000` Гц  
- `IND_BANDS = (2, 1, 1, 1, 2, 7, 13, 23)`

Крок по частоті (1 бін):

- `Δf = Fs / N = 40_000 / 1024 = 39.0625 Гц/бін`

### Межі смуг

> Формат бінів: `k_start .. k_end` (включно)  
> Ширина: кількість бінів у смузі

| Смуга | Біни (k) | Ширина (біни) | Діапазон, Гц    |
| -----:| -------- | -------------:| --------------- |
| 1     | 1..2     | 2             | 39.1 … 117.2    |
| 2     | 3..3     | 1             | 117.2 … 156.3   |
| 3     | 4..4     | 1             | 156.3 … 195.3   |
| 4     | 5..5     | 1             | 195.3 … 234.4   |
| 5     | 6..7     | 2             | 234.4 … 312.5   |
| 6     | 8..14    | 7             | 312.5 … 585.9   |
| 7     | 15..27   | 13            | 585.9 … 1093.8  |
| 8     | 28..50   | 23            | 1093.8 … 1992.2 |

### Підсумок покриття

- Загальна кількість бінів у 8 смугах: `Σ IND_BANDS = 50`
- Орієнтовний діапазон відображення: **≈ 39 Гц … 1.992 кГц**  

Nyquist при `Fs=40 кГц`: `f_Nyq = Fs/2 = 20 кГц`  
Верхня частина **~2…20 кГц** свідомо не відображається.  

## 📈 Продуктивність

#### Вихідні означення

- `T_cap` — час захвату ADC:
  - `T_cap = FFT_SIZE / Fs`
- `T_fft` — час виконання FFT + вікно (`fastfft.rfft(..., window=True)`)
- `T_core1` — час обробки кадра на **Core1** після отримання спектра

---

### Розподіл задач між ядрами

У проєкті використано 2 ядра RP2040, конвеєр “producer → consumer”:

- **Core0 (producer):** захват ADC (`FFT_SIZE/Fs`) + FFT (`fastfft.rfft()`), публікація `spectr_front` (memoryview)
- **Core1 (consumer):** агрегація смуг + AGC + рендер + `np.write()` (WS2812B)

> У поточній схемі `fastfft.rfft()` повертає `memoryview` на внутрішній буфер, тому **Core0 не може запускати наступний `rfft()`**, поки Core1 не завершив читання/обробку попереднього спектра (`spectr_busy` handshake).

---

### Синхронізація між ядрами (1-слотовий буфер без копії спектра)

Оскільки спектр — це `memoryview` на внутрішній буфер `fastfft`, **наступний виклик `rfft()` перезапише дані**, які Core1 може ще читати. Тому використовується handshake:

- `spectr_busy = True` — Core1 ще працює зі спектром поточного кадра  
- `spectr_busy = False` — Core0 може виконувати наступний `rfft()`

Core0 перед `rfft()` очікує, поки `spectr_busy` стане `False`.  
Core1 може виконуватись паралельно з Core0 **лише під час наступного захвату ADC**.  
Перед запуском наступного `rfft()` Core0 має дочекатись, щоб Core1 завершив роботу з попереднім `memoryview`.

---

### Оцінка періоду кадра для поточної 2-ядерної схеми

#### Основна оцінка періоду кадра

T_frame ≈ T_fft + max(T_cap, T_core1)

Інтерпретація:

- якщо T_core1 ≤ T_cap  →  T_frame ≈ T_cap + T_fft
- якщо T_core1 > T_cap  →  T_frame ≈ T_fft + T_core1

#### FPS

FPS ≈ 1 / T_frame

---

### Числовий приклад (FFT_SIZE=1024, Fs=40_000, T_fft≈35 ms, T_core1(min)≈20 ms, T_core1(max)≈65 ms)

1) Час захвату:
   T_cap = FFT_SIZE / Fs = 1024 / 40000 = 0.0256 s = 25.6 ms

2) Порівняння з Core1:
   max(T_cap, T_core1(min)) = max(25.6 ms, 20 ms) = 25.6 ms - мінімальне значення
   max(T_cap, T_core1(max)) = max(25.6 ms, 65 ms) = 65 ms - максимальне значення

3) Мінімальний період кадра:
   T_frame ≈ T_fft + max(T_cap, T_core1(min))
   
        ≈ 35 ms + 25.6 ms
        ≈ 60.6 ms

4) Максимальний період кадра:        
   T_frame ≈ T_fft + max(T_cap, T_core1(max))
   
        ≈ 35 ms + 65 ms
        ≈ 100 ms

5) FPS:
   FPS min ≈ 1 / 0.0606 ≈ 16.5 FPS
   FPS max ≈ 1 / 0.10 ≈ 10.0 FPS
